# LongMemEval Retrieval Benchmark — 2026-05-01

Branch: `bench/longmemeval-2026-05-01`

## Headline

**Smart pipeline (SmartRetrieval / ADR-090) beats raw HNSW** on the metrics that matter for end-user answer quality, at a fixed-k=10 retrieval budget. Wins at n=30 are amplified at n=500.

| Metric | RAW (n=500) | SMART (n=500) | Δ |
|---|---|---|---|
| Session R@10 | 100.0% | 100.0% | 0 |
| Session Top-1 | 100.0% | 100.0% | 0 |
| **Content@1** | 22.2% | **24.6%** | **+2.4 pp** |
| Content@3 | 35.8% | 34.6% | −1.2 pp |
| Content@10 | 43.6% | 43.2% | −0.4 pp |
| **MRR (content)** | 0.2967 | **0.3058** | **+0.0091** |
| Retrieval p50 | 0.02 ms | 5.79 ms | +5.77 ms |
| Retrieval p95 | 0.03 ms | 13.73 ms | +13.70 ms |

The smart pipeline's small p50/p95 increase (still well under 15 ms) is the cost of query expansion + RRF + recency + MMR + session round-robin running over the same HNSW index. Both strategies share embedding + index — the delta is fully attributable to the pipeline.

## Addendum 2026-05-01b — MiniLM (real semantic embedder) vs hash (bag-of-words)

After the runs above, an audit of `run-retrieval-bench.ts` revealed that the previous "embed" function was **not a learned encoder** — it was a deterministic hash of word unigrams + word bigrams + character 3-grams into a 384-dim space. This is bag-of-words, not semantic similarity. ADR-088's roadmap presumed `all-MiniLM-L6-v2` was the existing baseline; in reality there was no semantic encoder in the bench.

The bench was extended with `--embedder hash|minilm|bge-small|bge-base|bge-large` (commit forthcoming). MiniLM (`Xenova/all-MiniLM-L6-v2`, 384d, ONNX-quantized) was run at n=500 for both `raw` and `smart` strategies.

| Session R@10 | 100.0% | 100.0% | 100.0% | 100.0% |
| Session Top-1 | 100.0% | 100.0% | 100.0% | 100.0% |

Where token overlap matters less (`single-session-assistant`, `single-session-preference`), the encoders track each other.

### Why MiniLM lost — and what this means for the optimization roadmap

The bench's `Content@k` metric is **lexical substring match**:

```ts
// run-retrieval-bench.ts:scoreRun()
const matchesAnswer = (content: string): boolean => {
  const c = content.toLowerCase();
  return answerNorms.some(a => c.includes(a));
};
```

A retrieved chunk only counts as a "hit" if the **exact answer string** appears as a substring. Hash's character-3-gram and word-bigram features inadvertently maximize token-overlap — which is what this metric rewards. Dense semantic embeddings (MiniLM, bge-large) optimize for *semantic* similarity, which is a different objective: a chunk that paraphrases the answer ranks high in cosine similarity but scores 0 on substring match.

This invalidates the simplest reading of ADR-088's Tier 1 #1 ("swap MiniLM → bge-large for +5–15 pp"). A stronger semantic encoder optimizes the wrong objective for this metric. **The actual Tier-1 priorities reorder**:

1. **Tier 1 #3 (BM25 + dense via RRF) jumps to first.** BM25 IS lexical retrieval done properly — full-document scoring with TF-IDF and length normalization, not opportunistic token overlap. Combining BM25 (which the metric rewards) with dense (which captures semantic recall when keywords don't match) via RRF should beat both individually.
2. **Tier 1 #2 (semantic chunking) still matters.** Many `single-session-preference` answers (0% across both encoders) are mid-message; chunk boundaries determine whether the answer text lands in retrievable chunks at all.
3. **Tier 1 #1 (bge-large swap) deferred** until after BM25 hybrid lands. May still help on multi-hop categories, but expected gain is smaller than first thought on this lexical-match metric.

There is also a separate, larger question: **is `Content@k` the right metric for AgentDB?** LongMemEval's official evaluator (`evaluate_qa.py`) uses an LLM to grade a *generated answer* against the gold reference — not substring-match the retrieved chunks. The bench measures retrieval-only quality with a strict lexical proxy. Future work: wire `evaluate_qa.py` for an end-to-end (RAG + Haiku QA) score that credits semantic matches.
